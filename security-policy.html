Credit Split

Information Security Policy & Data Handling Procedures
Version 1.0  |  Effective date: March 2026  |  Classification: Internal / Regulator

1. Organisation Overview

Credit Split is a personal finance application that helps two people sharing a credit card account (a primary card holder and an additional card holder) reconcile shared spending. The app imports credit card transaction data, attributes each transaction to the correct person, and calculates outstanding balances between them.

Credit Split is operated as a small-scale consumer application. The development team maintains direct oversight of all infrastructure, code, and data access.

2. Scope

This policy covers all systems, data stores, and third-party services involved in operating Credit Split:

  • The Credit Split mobile application (React Native / Expo)
  • Supabase backend (authentication, database, edge functions)
  • Plaid API integration (transaction data aggregation)
  • All user data processed or stored by the above

4. Plaid Integration — Data Flow & Security

4.1 How Plaid data is used

Credit Split uses Plaid's Transactions product exclusively. The purpose is to import credit card transaction history so the app can attribute each spend and payment to the correct card holder. Specifically:

  1. Plaid Link — The user authenticates with their bank via Plaid's hosted Link interface (WebView). Credit Split never sees or handles bank login credentials.
  2. Token exchange — Plaid returns a public_token which is exchanged server-side (Supabase Edge Function) for an access_token. The access_token is stored in the database and is never sent to the client app.
  3. Transaction sync — A server-side Edge Function uses the access_token to call Plaid's /transactions/sync endpoint. Only the following fields are extracted and stored: transaction date, merchant/description, amount, and the originating card account (mapped to primary or secondary).
  4. No re-sharing — Plaid data is not shared with any third parties, used for advertising, sold, or aggregated across users.

4.3 Plaid access token handling

  • Access tokens are stored in the plaid_items table with Row-Level Security enabled
  • The client app never receives the access_token — it is only read by server-side Supabase Edge Functions
  • Client queries to plaid_items explicitly select only: id, institution_name, last_synced, status
  • When a user disconnects a bank, the access_token is deleted from the database

6. Access Control

  • User access — Email + password authentication. Users can only access data for accounts they are a member of (enforced by RLS policies at the database level).
  • Account sharing — A primary user generates a time-limited invite code (valid 7 days). The secondary user redeems it to join the shared account. No other sharing mechanism exists.
  • Developer access — Supabase dashboard access is restricted to the development team via Supabase organisation-level authentication. Database access requires the service role key.
  • No public endpoints — All API endpoints require a valid JWT. Edge Functions verify both authentication and account membership before processing requests.

7. Data Lifecycle & Deletion

User clears dataAll transactions and payment attributions for their account are deleted from the database. Account and membership remain.
User disconnects bankPlaid access_token and card mappings are deleted. Previously synced transactions remain (user can clear these separately).
User deletes accountCascading delete: all transactions, allocations, invite codes, Plaid items, card mappings, account membership, and the account itself are permanently removed. The auth user record is also deleted. User is signed out.
Invite codesExpire after 7 days. Old unused codes are deleted when a new code is generated.

9. Incident Response

  1. Detection — Monitor Supabase logs and Edge Function error rates for anomalous activity.
  2. Containment — Revoke compromised Plaid access tokens via Plaid API; rotate Supabase service keys if needed; disable affected user accounts.
  3. Notification — Affected users notified within 72 hours of confirmed breach, consistent with GDPR requirements.
  4. Recovery — Restore from Supabase point-in-time backups if data integrity is compromised.
  5. Review — Post-incident review to identify root cause and implement preventive measures.

10. Compliance

  • GDPR — Users can view, export, and delete all their data at any time through the app's Settings screen. No data is processed beyond what is necessary for the stated purpose (reconciling shared card spending).
  • Plaid End User Privacy Policy — Credit Split's use of Plaid data is limited to the Transactions product for the sole purpose described in this document.
  • Data minimisation — Only transaction fields required for reconciliation are stored. Full Plaid response objects are not persisted.

11. Policy Review

This policy is reviewed and updated when material changes are made to the application's data handling, infrastructure, or third-party integrations. The current version reflects the application architecture as of March 2026.
