Data Retention & Deletion Policy

Credit Split — Personal Finance Reconciliation Application
Document ID: CS-DRP-001  |  Version 1.0  |  Effective: March 2026  |  Owner: Tom Sharples  |  Next review: March 2027

1. Purpose

This policy defines how Credit Split retains, deletes, and disposes of user data. It ensures compliance with applicable data privacy laws — including the UK GDPR and the Data Protection Act 2018 — and establishes that data is kept only as long as necessary to provide the service.

Core principle: User data is retained only for as long as the user actively uses the service. Users have full control to delete any or all of their data at any time, with immediate effect.

2. Scope

This policy applies to all user data stored by Credit Split, including:

  • Authentication records (Supabase Auth)
  • Account and membership data
  • Credit card transaction records (from CSV upload or Plaid sync)
  • Payment attribution decisions
  • Plaid connection data (access tokens, card mappings)
  • Invite codes

4. User-Initiated Deletion

Credit Split provides three self-service deletion mechanisms, all accessible from the Settings screen in the app:

4.1 Clear All Data

  • What it deletes: All transactions and payment attributions for the account
  • What it preserves: Account, membership, and Plaid connections
  • Confirmation: Requires explicit user confirmation via alert dialog
  • Implementation: clearAccountData() function — deletes from payment_allocations and transactions tables, resets card member names on the account

4.2 Disconnect Bank

  • What it deletes: Plaid access token and all card-to-person mappings for that connection
  • What it preserves: Transactions that were already synced (user can clear these separately)
  • Confirmation: Requires explicit user confirmation via alert dialog
  • Implementation: disconnectPlaidItem() function — deletes from plaid_card_mappings and plaid_items tables

4.3 Delete Account

  • What it deletes: Everything — all transactions, attributions, Plaid items, card mappings, invite codes, account membership, the account itself, and the authentication record
  • What it preserves: Nothing. Complete and irreversible removal.
  • Confirmation: Requires two explicit confirmations via sequential alert dialogs
  • Implementation: delete_account server-side database function with cascading deletes across all tables. User is signed out immediately after deletion.

5. Deletion Completeness

5.1 Database deletion

All deletion operations are hard deletes — rows are permanently removed from the PostgreSQL database, not soft-deleted or flagged. There is no recycle bin or undo mechanism.

5.2 Backups

Supabase Cloud maintains automated point-in-time recovery (PITR) backups for disaster recovery. Deleted data may exist in backups for up to 7 days (Supabase Pro plan retention window). Backups are encrypted at rest and accessible only to Supabase infrastructure — not to Credit Split administrators or users. After the backup retention window, deleted data is permanently unrecoverable.

5.3 Plaid-side data

When a user disconnects their bank, Credit Split deletes the access token from its database. Plaid may retain its own copy of the connection and transaction data according to Plaid's privacy policy. Users can request deletion directly from Plaid via their Plaid Portal.

5.4 Client-side data

The Credit Split mobile app stores no data on the device. Session tokens are held in-memory only and are cleared on sign-out or app restart. There is no local database, cache, or file storage to purge.

8. Compliance Verification

The deletion mechanisms described in this policy are enforced programmatically in the application's source code and database schema (foreign key cascades). They are not discretionary or manual processes. Specifically:

  • clearAccountData() — deletes allocations and transactions, resets card member names
  • disconnectPlaidItem() — deletes card mappings and Plaid item record
  • delete_account() — server-side SQL function that cascades deletion across all tables and removes the auth user record
  • Foreign key constraints with ON DELETE CASCADE ensure referential integrity — child records cannot be orphaned
