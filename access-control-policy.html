Access Control Policy

Credit Split — Personal Finance Reconciliation Application
Document ID: CS-ACP-001  |  Version 1.0  |  Effective: March 2026  |  Owner: Tom Sharples  |  Classification: Internal / Regulator

1. Purpose

This policy defines the access control requirements for all systems, data, and infrastructure that comprise the Credit Split application. Its objective is to ensure that access to production assets and sensitive data is limited to authorised individuals and services on a least-privilege basis.

2. Scope

This policy applies to:

  • The Credit Split mobile application and its source code
  • Supabase project infrastructure (database, authentication, edge functions, dashboard)
  • Third-party integrations (Plaid API)
  • All end-user accounts
  • All personnel with administrative or development access

3. Guiding Principles

Least Privilege: Every user, service, and system component is granted the minimum level of access necessary to perform its function.
Role-Based Access: Access rights are assigned based on defined roles, not on an individual basis.
Defence in Depth: Access is enforced at multiple layers — application, API, and database — so that no single control failure results in unauthorised access.
Separation of Concerns: Client-side code cannot access server-side secrets. Administrative functions are separated from end-user functions.

5. Database Access Control (Row-Level Security)

All five application tables have PostgreSQL Row-Level Security (RLS) enabled. RLS policies enforce that users can only access rows belonging to accounts they are a member of.

TableRLS PolicyEffect
accountsAuthenticated users only; membership checkedUsers can only read/modify their own account
account_membersAuthenticated users only; scoped to own accountUsers can only see members of their account
transactionsAuthenticated users only; scoped to own accountUsers can only read/write transactions for their account
payment_allocationsAuthenticated users only; scoped to own accountUsers can only manage allocations for their account
invite_codesAuthenticated users only; scoped to own accountUsers can only see/create codes for their account

Key enforcement mechanism: The SQL function is_account_member(aid uuid) is used in RLS policies to verify that auth.uid() has a row in account_members for the given account. This check runs at the database level on every query — it cannot be bypassed by the client.

7. API & Edge Function Access Control

All Supabase Edge Functions enforce a two-step authorisation check before processing any request:

  1. Authentication — The JWT from the request's Authorization header is verified via supabase.auth.getUser(). Invalid or missing tokens are rejected.
  2. Account membership — The authenticated user's ID is checked against account_members for the requested account_id. Non-members are rejected with a 403 error.

This pattern is implemented in the shared verifyAccountAccess() function used by all Edge Functions.

9. Invite Code Access Control

Sharing access to a Credit Split account is controlled through single-use, time-limited invite codes:

  • Only the primary role can generate invite codes
  • Codes are 8-character random alphanumeric strings
  • Codes expire after 7 days
  • Each code can be redeemed once — redemption is atomic (database function with row-level locking)
  • Generating a new code invalidates all previous unused codes for that account
  • Redemption grants the secondary role only
